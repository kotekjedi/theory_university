## Чеклист и как его составить?
Короткий список проверки функциональности чего-либо.

Структурирует тестовую информацию, формирует порядок действий, повышает скорость вхожения нового тестировщика, позволяет заменять тестировщиков, можно показывать разработчикам для оценки прогресса.

Правила составления:
* один пункт - одна операция
* пункты в утвердительной форма
* оптимальный размер не более 20 пунктов

## Написать чеклист на кружку
1. У кружки есть ручка
2. У кружки есть дно
3. В кружку можно налить жидкость
4. Если в кружку налить жидкость, она не взрывается
5. Если в кружку налить жидкость меньше максимального объема, она не вытекает
6. Если перевернуть кружку, налитая жидкость вытекает
7. Если поместить кружку в жидкость, она не взрывается


## Написать чеклист на вилку
1. Можно держать в руке
2. Имеет хотя бы 2 зубца
3. Зубцы протыкают мягкий материал
4. Если взять за рукоятку, вилка не ломается
5. Если взять в руку, не взрывается
6. Если поместить в жидкость, не растворяется
7. Можно поднять одной рукой


## Как выбрать приоритет бага?
Зависит от процента пользователей, которые с ним сталкиваются и какую часть функционала затрагивает.

* Критичный - обычно больше 50%
* Major
* Minor
* Базовый

Серьезность дефекта:
* Блокирующий - блокирует тестирование или часть возможных пользовательских сценариев.
* Критический - нарушение ключевой логики, появление уязвимостей...
* Значительный - не работает важная часть приложения, но есть способ ее обойти, воспользовавшись другой частью функционала.
* Незначительный - визуальные ошибки, ошибки со средствами доступности, похожие цвета
* Минор - очень незначительные, отличие размеров шрифтов на 1 кегль...

## Что такое тестовое окружение?
Условия в которых проводится тестирование
* среда разработки: тестовые данные,  тестовые db, тестовый бэкенд
* среда тестирования: здесь работают тестировщики, проверяют функционал, запускаются автотесты, обычно небольшой тестовый бэкенд и db,
* интеграционная среда: тестирование работы взаимодействующих модулей
* предпродовое окружение: уже реальный бэкенд, проверка, что все работает
* продакшн среда: все используется, мы наблюдаем 

Фазы тестирования:
* пре-альфа: на стадии разработки
* альфа: тестирование первой версии, последняя только для разработчиков
* бета: либо привлекаем часть пользователей, либо специальные тестировщики
* стабилизация: исправляем баги, возвращаемся в бету
* релиз сборка кандидата: скорее всего будет использоваться, проходит все необходимые проверки
* релиз

## Функциональное и нефункциональное тестирование
### Функциональное тестирование
Функциональное тестирование - тестирование, направленное на проверку соответствий функциональных требований ПО к его реальным характеристикам. Основная задача - подтверждение того, что продукт обладает всем функционалом, требуемым заказчиком.

### Нефункциональное тестирование
Определяем как работает ПО

| Название | Описание | пример |
|---|---|---|
| Надежность | реакция системы на непредвиденные ситуации | 1) приложение отключилось от сервера, должно обработать ошибку, показать информацию и не упасть, 2) поддержка разных браузеров |
| Производительность | работоспособность системы под разными нагрузками, оценка использования ресурсов | замер времени запуска, кэширования |
| Удобство использования | характеризует систему с точки зрения удобства использования конечного пользователя | проверяем соответствие ожиданиям дизайнеров, понятность интерфейста для пользователя (что делать в разных условиях) добавление подсказок | 
| Масштабируемость | требования к горизонтальному и вертикальному масштабированию, обычно к бэкенду | проверяем работу с несколькими серверами |
| Безопасность | защищенность пользовательских данных, дорогие уязвимости, должно происходить на самых ранних стадиях разработки |  | XSS - подмена реального контента вредоносным кодом, SQL-инъекции, пути в логах..., лишняя информация в отрытом доступе |
| Портируемость | переносимость приложения на разные платформы | игра со сборкой и корректной работой на разных ОС, адаптивность сайта |
| Установка | проверка успешности установки, настройки, обновления и удаления ПО | снижаем риск потери данных, лишения работоспособности |
| Отказ и восстановление | исследования на предмет восстановления после ошибок сбоев, оценивание реакции  защитных свойств приложения | что будет после восстановления бэкенда |

## Из чего сосотоит HTTP-запрос
Позволяет обмениваться данными между клиентом и сервером.

__Запрос__
* Метод: GET, PUSH, PUT, DELETE, PATCH, HEAD, CONNECT, TRACE, OPTIONS
* Путь
* Версия протокола
* Хост
* Заголовки

__Ответ__
* Версия
* Код статуса: 
  - 1xx информационные
  - 2xx успешные ответы
  - 3xx перенаправления
  - 4xx ошибка с запросом
  - 5xx ошибка на сервере
* Сообщение статуса
* Заголовки
* Тело

Обычно тело ответа с сервера - JSON, это REST (Representational State Transfer)

Также бывают SOAP (Simple Object Access Protocol) - отправляем XML.

И RPC (Remote Procedure Call) - запускаем процедуры на клиенте

## Почему нельзя тестировать на проде?
* наличие багов может помешать нормальной работе приложения
* могут быть потеряны / испорчены реальные данные пользователй
* больше времени на поиск причины ошибки
* отрицательное впечатление пользователей, уменьшение их количества
* возможно баги редкоиспользуемого функционала не будут найдены
* если есть проблема в безопасности, то может быть получен доступ злоумышленниками, могут утечь данные пользователей
* если проблема с установкой / обновлением, пользователи не смогут использовать, будут потеряны клиенты

## Зачем нужны автотесты?
* ускоряют процесс тестирования, автоматизируют большую часть проверок
* избавляют от рутины путем автоматизации проверок
* позволяют найти баги на ранних этапах

## Из чего состоит багрепорт?
* Заголовок - суть проблемы, подробно и конкретно
* Подробное описание - с деталями, возможно с причинами бага
* Шаги для воспроизведения - действия, приводящие к багу, без излишек, конечный результат и ожидаемый результат
* Файлы, скриншоты c указанием локации и конкретного места ошибки, можно видео воспроизведения
* Критичность
* Приоритет
* Окружение - версия ОС, разрядность, процессор, тип устройства, версии зависимостей, версии интерпретатора языка, браузер, плагины

## Особенности тестирования мобильных приложений
* много разных устройств, разных настроек, условий, местоположений, подключений
* всегда нужно проверять на разных устройствах, разных систем
* приложение должно корректно работать и при хорошей связи, и при плохой
* если предполагаются разные страны, то нужно учитывать пересечение границы, из-за этого могут меняться  правила работы с данными
* если предполагаются разные страны, то нужно учитывать пересечение границы, из-за этого могут меняться  правила работы с данными

Для тестирования можно различных условий можно:
* запустить тестировщика кататься в метро и использовать приложение
* воспользоваться debug-proxy, например, Charles или Progress. 
  - симуляция разных условий запросов
  - обработка, фильтрация запросов
  - симуляция плохой связи
* у каждой платформы свои guideline-ы, описывают правила расположения кнопок, визуальных элементов, причем для разных классов устройств эти правила разные.
* PUSH-уведомления
  - либо через push-сервис (Google Provide Messaging, Huawei Push Serivce, Apple Push Notification service), нужно следить за приоритетом
  - либо для открытых приложений можно напрямую отправлять

Может использоваться как интерфейс, например, плеер и экран звонков в мессенжере.

* сбор статистики для улучшения интерфейса и контроля падений приложения

## Особенности тестирования WEB приложений
__Браузерный движок__:
* Blink, ответвление от kmWebKit - движок Сhromium, Google Chrome 28+, Edge 79+, Opera 15+, Vivaldi
* Gecko, открытый движок Mozilla - Firefox, Thunderbird, SeaMonkey, ...
* WebKit - движок Safari
* KHTML, разработан в рамках KDE, основа для WebKit - браузер Konqueror

У каждого движка свои методы работы с тегами и JS, поэтому нужно тестировать на разных движках и разных версиях.

Нужно тестировать для разных размеров экранов, на сенсорных экранах, при разных ориентациях.

__Инструменты__:
* Dev Tools в браузере, страница, консоль, стили, все запросы
* PostMan для тестирования API (запросы, ответы, базовые автотесты)

#### Типичные уязвимости
* XSS, генерируется опасный для клиента код на странице.
* XCRF, переходим на вредоностную страницу и передаем cookie на него / get-запросы 
* инъекции, добавляем свой код в запрос, можем получать / менять данные. (Напрмиер, PHP или SQL)
* обход авторизации
